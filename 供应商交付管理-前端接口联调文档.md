## 一、菜单结构

### Admin 后台

```
📦 供应商交付管理
├── 📋 请购单管理                → /admin/purchase/request
├── 📋 采购订单管理              → /admin/purchase/order
├── 📋 交付确认记录              → /admin/delivery/confirmation
└── 📊 交付时效分析              → /admin/delivery/timeliness
```

### 供应商端

```
📦 我的交付
├── 📋 我的采购订单              → /vendor/delivery/myPurchaseOrders
├── 📋 我的交付记录              → /vendor/delivery/myConfirmations
└── 📊 我的交付统计              → /vendor/delivery/myTimeliness + /myStats
```

## 二、通用说明

### 响应格式

```json
{
  "code": 200,
  "msg": "加载成功",
  "data": { ... }
}
```

### 分页参数（所有列表接口通用）

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| p | int | 否 | 页码，默认 1 |
| l | int | 否 | 每页数量，默认 10 |

### 分页响应结构（PageTO）

```json
{
  "total": 100,
  "list": [ ... ]
}
```

### 时间参数格式

所有 `beginTime` / `endTime` 参数格式为 `yyyy-MM-dd`，例如 `2025-02-01`

### 4.6 供应商回复日期确认（最新）

```
POST /vendor/delivery/reply-date
```

**Content-Type：** `application/json`  
**鉴权：** `Authorization: Bearer <token>`  
**说明：** 后端通过 `@VendorCode` 从登录态自动注入供应商编码，前端无需传 `vendorCode`

**请求体：**

```json
{
  "poId": 123,
  "replyType": 2,
  "replyPlanList": [
    {
      "replyDate": "2026-03-10",
      "deliveryQty": 5000
    },
    {
      "replyDate": "2026-03-20",
      "deliveryQty": 10000
    }
  ]
}
```

**字段说明：**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| poId | Long | 是 | 采购订单明细ID |
| replyType | Integer | 是 | 回复类型：`1` 一次性交货，`2` 分批次交货 |
| replyPlanList | Array | 是 | 回复计划列表 |
| replyPlanList[].replyDate | String | 是 | 回复日期，格式 `yyyy-MM-dd` |
| replyPlanList[].deliveryQty | Number | 是 | 交货数量，必须大于 `0` |

**业务规则：**

1. `replyType = 1` 时，`replyPlanList` 必须且只能 1 条。  
2. `replyType = 2` 时，`replyPlanList` 至少 2 条。  
3. 每条计划的 `replyDate`、`deliveryQty` 均不能为空。  
4. 每条计划 `deliveryQty` 必须大于 `0`。  
5. `replyPlanList` 总数量不能大于采购订单数量。  
6. 系统会将计划中最大的 `replyDate` 写入原有字段 `vendor_reply_date`。  
7. 系统会将 `replyType` 写入 `vendor_reply_type`，将计划列表 JSON 写入 `vendor_reply_plan`。  

**响应示例：**

成功：

```json
{
  "code": 200,
  "msg": "回复日期确认成功",
  "data": true
}
```

常见失败：

```json
{"code":50060,"msg":"采购订单ID不能为空","data":false}
{"code":50060,"msg":"回复类型错误","data":false}
{"code":50060,"msg":"一次性交货只能提交一条回复计划","data":false}
{"code":50060,"msg":"分批次交货至少提交两条回复计划","data":false}
{"code":50060,"msg":"回复计划中的日期和交货数量不能为空","data":false}
{"code":50060,"msg":"交货数量必须大于0","data":false}
{"code":50060,"msg":"回复计划总数量不能大于采购数量","data":false}
{"code":500,"msg":"采购订单不存在","data":false}
{"code":500,"msg":"无权操作此采购订单","data":false}
```

**curl 联调示例：**

```bash
curl -X POST "http://<host>/vendor/delivery/reply-date" \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer <token>" \
  -d "{\"poId\":123,\"replyType\":2,\"replyPlanList\":[{\"replyDate\":\"2026-03-10\",\"deliveryQty\":5000},{\"replyDate\":\"2026-03-20\",\"deliveryQty\":10000}]}"
```

## 五、数据库变更 SQL

### 5.1 采购订单表新增字段（回复计划）

```sql
ALTER TABLE t_purchase_order
  ADD COLUMN vendor_reply_type TINYINT(1) NULL COMMENT '供应商回复类型(1:一次性交货 2:分批次交货)' AFTER vendor_reply_date,
  ADD COLUMN vendor_reply_plan LONGTEXT NULL COMMENT '供应商分批回复计划(JSON)' AFTER vendor_reply_type;
```

---

### 5.2 新增索引 SQL

```sql
-- 请购单：按请购日期筛选
ALTER TABLE `t_purchase_request` ADD INDEX `idx_request_date` (`request_date`) USING BTREE COMMENT '请购日期索引';

-- 采购订单：按请购单号搜索 + 按制单时间筛选
ALTER TABLE `t_purchase_order` ADD INDEX `idx_app_code` (`app_code`) USING BTREE COMMENT '关联请购单号索引';
ALTER TABLE `t_purchase_order` ADD INDEX `idx_make_time` (`make_time`) USING BTREE COMMENT '制单时间索引';

-- 交付确认：按创建时间筛选
ALTER TABLE `t_delivery_confirmation` ADD INDEX `idx_create_time` (`create_time`) USING BTREE COMMENT '创建时间索引';

-- 交付时效统计：按创建时间筛选
ALTER TABLE `t_delivery_timeliness` ADD INDEX `idx_create_time` (`create_time`) USING BTREE COMMENT '创建时间索引';
```

## 六、回填规则说明

- `vendor_reply_date`：回填计划中的最大日期，用于兼容现有列表/风险判断逻辑。
- `vendor_reply_type`：记录回复类型（1/2）。
- `vendor_reply_plan`：按日期排序后的计划明细 JSON。
